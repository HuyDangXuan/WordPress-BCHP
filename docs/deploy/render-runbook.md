# HV-Travel Render Runbook

This runbook prepares the project for a Render deployment without performing a live deploy from this repo phase.

## Service Matrix

| Service | Render type | Runtime | Public | Persistent disk | Responsibility |
| --- | --- | --- | --- | --- | --- |
| WordPress | `web` | Docker | Yes | `/var/www/html/wp-content/uploads` | Storefront, admin, WooCommerce, theme, plugin |
| booking-payment-service | `pserv` | Docker | No | No | Booking sync, ZaloPay QR callback, revenue report |
| mysql | `pserv` | Image `mysql:8.0` | No | `/var/lib/mysql` | WordPress and WooCommerce database |
| mongodb | `pserv` | Image `mongo:7.0` | No | `/data/db` | `bookings`, `payments`, `payment_events` |

Only WordPress is public. MySQL, MongoDB, and `booking-payment-service` stay on Render private networking.

## Env Matrix

WordPress receives:

- `WORDPRESS_DB_HOST` from the private MySQL hostport.
- `WORDPRESS_DB_NAME=wordpress`.
- `WORDPRESS_DB_USER=wordpress`.
- `WORDPRESS_DB_PASSWORD` from the MySQL service env.
- `WP_DEBUG=false`.
- `PAYMENT_SYNC_SECRET` generated by Render.
- `BOOKING_SERVICE_ENDPOINT` entered manually after the private service host is known.
- `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS` entered manually if email is part of the demo.

Booking/payment service receives:

- `PORT=8787`.
- `MONGO_URI` entered manually, for example `mongodb://<mongodb-private-host>:27017/hv_travel`.
- `PAYOS_CLIENT_ID`, `PAYOS_API_KEY`, `PAYOS_CHECKSUM_KEY` entered manually.
- `ZALOPAY_APP_ID`, `ZALOPAY_KEY1`, `ZALOPAY_KEY2` entered manually from ZaloPay Merchant Portal.
- `ZALOPAY_ENV=sandbox` for test mode or `production` for live mode.
- `ZALOPAY_CALLBACK_URL` entered manually, for example `https://<service-domain>/api/payments/zalopay/callback`.
- `PAYMENT_SYNC_SECRET` referenced from WordPress.
- `WORDPRESS_CONFIRM_ENDPOINT` entered manually, for example `https://<wordpress-domain>/wp-json/op-travel/v1/payment-confirm`.

Render Blueprint files do not interpolate strings, so endpoint URLs with paths remain manual values.

## Deploy Steps

1. Push the release branch or tag to the Git remote connected to Render.
2. Create a Blueprint from `render.yaml`.
3. Fill every `sync: false` secret or endpoint in the Render dashboard.
4. Confirm disks are attached before first real demo data entry.
5. Deploy private services first, then WordPress.
6. Open WordPress admin, install WooCommerce if needed, activate `OP Travel Shop` and `OP Travel Core`.
7. Run the OP Travel demo seeder.
8. Set permalinks to refresh rewrite rules if `/tours/` or REST routes return 404.

## Backup And Restore

- MySQL: take `mysqldump` before deploys that change WordPress or WooCommerce behavior.
- MongoDB: take `mongodump` before webhook/payment testing on shared demo data.
- Uploads: keep the Render disk snapshot, but use application-level backups for files needed in the defense demo.
- Restore order: MySQL first, MongoDB second, uploads third, then smoke test.
- Do not rely on raw disk snapshot restore for custom database recovery unless no logical dump exists.

## Rollback

- Roll back by selecting the last known-good Git tag or commit in Render.
- Do not hotfix files through the Render shell.
- After rollback, verify env values were not changed during the failed deploy.
- Run the post-deploy smoke checklist before handing the demo link back to the team.

## Post-Deploy Smoke

Run these checks after every deploy or rollback:

- Open homepage and confirm the custom theme loads.
- Open `/tours/` and confirm archive/filter content is visible.
- Open `/tours/premium-hoang-hon-phu-quoc/` and confirm booking metadata is visible.
- Create a test order and confirm cart, checkout, and thank-you page work.
- Confirm `POST /wp-json/op-travel/v1/payment-confirm` does not return 404.
- Confirm `GET /health` on the service from a private-network shell.
- Confirm MongoDB has `bookings`, `payments`, and `payment_events` after a webhook fixture.
- Review WordPress/PHP logs and service logs before demo.
